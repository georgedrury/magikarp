# Magikarp benchmark — diverse / falsification suite (2026-06-23)

A second suite, built to **break the thesis** rather than confirm it: anti-cases where minimal is *wrong*, a never-cut safeguard under minimalism pressure, an ambiguous spec, a build with no future payoff, a brownfield change, two non-TypeScript languages — plus a **correctness gate** (an independent judge on every build). It is not a clean sweep, and that is the point.

**Model:** claude-opus-4-8 · **Samples:** 3 per cell · 6 tasks × 2 arms
**Measurement:** deterministic line/file diffs for build/extend/blast; an independent LLM judge for correctness and for the guard/surgical properties (these are judged, not deterministic — a stated limitation).
**Raw data:** [2026-06-23-v2-diverse-raw.json](2026-06-23-v2-diverse-raw.json)

## What each result means

- **07 — the red flag (real, and worth acting on).** Magikarp built about half the control's code (57 vs 112 LOC) and stayed injection-safe and correct — but in **1 of 3 samples its leanest build (36 LOC) shipped incomplete boundary validation** (missing required-field / type checks). Input validation at a trust boundary is on the never-cut list, so that is a discipline shortfall, not a saving. At n=3 it is one miss, but it is exactly the failure mode this task was built to find: under the minimalism gradient, a never-cut item got shaved. **Action item:** harden the never-cut language in `core/magikarp.md` (and re-test 07 at higher n to confirm the rate).

- **08 — the cleanest win.** Both arms *surfaced* the ambiguity (the judge passed both), but the discipline changed the build: Magikarp shipped a 27–39 LOC in-process memo and explicitly flagged every unspecified knob; the control documented its assumptions **and still built 201–239 LOC** of TTL/LRU/eviction config the ticket never asked for. "Surface, don't guess" produced surface *and defer*, not surface *and build it all anyway*.

- **10 — modest win in Java.** The over-abstraction archetype's home language: Magikarp built ~20% less and absorbed the second name-format in ~20% fewer lines. The control did *not* build a giant factory here (the model is capable), so the gap is modest — honestly reported as such. Supports the language-agnostic claim.

- **09 — no win, slight loss.** On a one-line health check there is nothing to over-engineer, so the discipline's small overhead (a comment / marker) made Magikarp marginally *heavier* (18 vs 14 LOC). Both correct. A benchmark that only ever shows wins would be suspect; this is a task where there was nothing to save.

- **11 — tie.** Both arms made the surgical 2-line change with no "while I'm here" refactoring. The reflex simply did not fire for the control on a change this small; a more tempting brownfield task might separate the arms. Reported as a tie.

- **06 — invalid (a harness bug, not a result).** The task requires wiring a shared helper into two *existing* call sites, but the harness bars filesystem access (to prevent the contamination that invalidated an earlier run), so the agents had no call sites to edit and correctly built the function while flagging they could not wire it. Both arms "failed" the property for the same structural reason. **Fix:** supply the two call-site files as a baseline (as the brownfield task does) and re-run. Until then, the under-engineering guard is unmeasured.

## Honest reading

- Where there is real over-engineering to resist (07, 08, 10), the discipline cut build size **20–84%** with **no loss of correctness**.
- Where there is nothing to resist (09 trivial, 11 tiny change), it ties or adds a hair — correctly, there was nothing to save.
- The **correctness gate held at 100%** on every valid task for both arms: minimal here did not mean broken.
- But the suite found a **genuine never-cut shortfall (07)** — the most important result, because it is the failure mode the whole "Splash, not slop" promise exists to prevent. The discipline is not yet bulletproof on never-cut under pressure.

## Limitations

- n=3 (directional; the 07 shortfall is one miss of three — confirm at higher n before drawing a rate).
- Single model (claude-opus-4-8). The control is a capable model that does not always over-engineer, which compresses some gaps honestly.
- The guard/correctness verdicts are **LLM-judged**, not deterministic, unlike the line-diff measures.
- Task 06 is invalid pending a harness fix.
- The first suite ([2026-06-23.md](2026-06-23.md)) predates the correctness gate; these are separate runs.

# Magikarp benchmark — 2026-06-23

> **Superseded for the headline by [2026-06-24-definitive.md](2026-06-24-definitive.md)** — a unified n=6, multi-model, correctness-gated run. This page is the original Opus n=3 run, kept for history.

**Model:** claude-opus-4-8 (workflow subagents, inherited)
**Samples:** 3 per cell · 5 tasks × 2 arms × 3 = 30 cells (60 agent runs: build + extend)
**Measurement:** deterministic. Each agent returns a structured file set; extension cost is a multiset line-diff (non-blank lines added + removed, files touched/created) between the build output and the extended output. No agent grades itself.
**Raw data:** [2026-06-23-raw.json](2026-06-23-raw.json)

## Arms
- **Control** — a capable engineer with no skill and no filesystem access; implements purely from the ticket.
- **Magikarp** — the same, with the behavioural core (`core/magikarp.md`) embedded in the prompt as an always-on discipline.

The only difference between arms is the presence of the core.

## Procedure
1. **Build** — the agent implements the spec (and never sees the extension).
2. **Extend** — a fresh agent receives the build's code plus a withheld, unplanned requirement and implements it.
3. We measure the step-2→step-3 change — the **extension cost**. Lower is the argument.

## Headline — extension cost (lines changed to absorb the unplanned requirement)

| Task | Pattern | Control | Magikarp | Reduction |
|---|---|--:|--:|--:|
| 01-config | config nobody asked for | 37 | 21 | **42%** |
| 02-utility | abstraction for one caller | 54 | 36 | **34%** |
| 03-error | a case the spec never mentions | 96 | 47 | **51%** |
| 04-badge | structure for one use (UI) | 114 | 46 | **60%** |
| 05-payments | the "abstraction was earned" honesty task | 740 | 180 | **76%** |

**Typical reduction ≈ 55%** (geometric mean of per-task ratios); median 51%. Magikarp absorbed the unplanned change in roughly half the changed lines on every task, and the direction held across all three samples of every task.

## Build cost (a speculation proxy — code written for the *original* spec)

| Task | Control | Magikarp |
|---|--:|--:|
| 01-config | 36 LOC / 1 file | 19 LOC / 1 file |
| 02-utility | 34 LOC / 2 files | 22 LOC / 1 file |
| 03-error | 29 LOC / ~1 file | 18 LOC / 1 file |
| 04-badge | 81 LOC / 3 files | 46 LOC / 2 files |
| 05-payments | **423 LOC / 6 files** | 70 LOC / 1 file |

The control arm consistently wrote more code for the same spec — most starkly on 05, where it built a six-file provider architecture for a one-function Stripe charge.

## Per-task notes

- **03-error (the never-cut boundary).** The control arm added speculative error handling for a checked-in file that *cannot* fail, then had to rewrite it (avg 2.3 constructs removed) when the *real* failure path — a remote fetch — arrived. Magikarp added no speculative handling at build, then added genuine handling at extension, exactly where an error path now exists. The discipline kept the never-cut case and dropped the speculative one — which is the whole point of that task.

- **05-payments (the honesty task — and the surprise).** This task was *designed* to be where speculation wins: the withheld Adyen requirement is precisely what a provider abstraction anticipates, so a pre-built provider interface should absorb it cheaply and beat Magikarp. It didn't. The unconstrained "production-grade" control build was not a tidy earned abstraction but a 5–7 file, 330–556 LOC architecture (consistent across all three samples), and absorbing Adyen cost it ~740 changed lines across ~11 files. Magikarp's direct 70-LOC Stripe implementation evolved into a two-provider router in ~180 lines.
  - **Honest caveat:** this measures what an unconstrained capable agent *actually builds* when told to ship production quality — and the reflex is sprawl, not a clean interface. A deliberately minimal speculative provider interface might fare better than this control did. The finding is not "speculation can never pay off"; it is "the real over-engineering reflex, even on the task built to favour it, produced the highest extension cost of all."

## Validity

- **A first run was discarded.** The benchmark agents run inside the Magikarp repo. In the first run, ~40% of *control* builds discovered the repo's own `rules/` and `core/` files and adopted the `*splash*` convention — contaminating the control arm. This run fixes that: every agent is barred from filesystem access and implements solely from the ticket, with the core supplied to the Magikarp arm in-prompt. **Control cleanliness was verified: 0/15 control cells reference `*splash*`** (vs 6/15 in the discarded run).
- **Determinism.** Generation is stochastic (the agents); measurement is not (a multiset line-diff computed in code).
- **Limitations.** Single model (claude-opus-4-8); N=3 per cell — directional, not a significance study, though all five tasks agree in direction with low variance. "Abstractions" are approximated by file and LOC counts, not AST parsing. The line-diff is a multiset approximation, applied identically to both arms. One language (TypeScript) throughout, to keep diffs comparable.

## Conclusion

On every task — including the one designed to favour speculation — the Magikarp arm wrote less code for the original spec and absorbed the unplanned requirement in roughly half the changed lines (42–76% fewer; ~55% typical). The headline claim holds: **a minimal build that defers speculative scope costs less to extend than a speculative one, because most speculation is the wrong shape.**
